# Paybook 공동가계부 설계

## 문제 정의

Paybook은 부부가 한 달 단위로 공동 지출을 빠르게 기록하고 확인하는 개인용 웹앱이다. 사용자는 로그인 없이 환경변수로 설정한 공유 PIN으로 접근하고, 누가 언제 어떤 용도로 얼마를 썼는지 남긴다. 최종 배포 형태는 Docker 이미지이며 Proxmox에서 컨테이너로 실행한다. 정산, 카드 연동, 영수증 OCR, 예산 관리는 MVP 범위에서 제외한다.

## 성공 기준

- 첫 사용 시 두 사람의 이름을 설정할 수 있다.
- PIN 입력 후 이번 달 전체 합계와 사람별 합계를 볼 수 있다.
- 한 화면에서 오늘 날짜 기본값으로 지출을 빠르게 추가할 수 있다.
- 지출 입력 시 사람은 두 이름 중 하나를 버튼/콤보박스 형태로 선택한다.
- 용도 입력 시 최근/자주 사용한 용도 5개를 보여주고, 사용자는 추천 항목을 선택하거나 자유 텍스트로 입력할 수 있다.
- 모바일 기본 화면은 지출 입력을 주 용도로 두고, 오른쪽 위 `상세내역` 버튼으로 월별 내역 화면에 들어간다.
- 상세 내역 화면에서는 날짜순 내역을 확인하고, 다른 달을 선택해 살펴보며, 잘못 입력한 항목은 삭제할 수 있다.
- 개인용 외부 URL 배포를 전제로 하되, PIN 없이는 장부를 볼 수 없다.

## 사용자 흐름

1. 배포자는 컨테이너 환경변수로 PostgreSQL 접속 정보와 공유 PIN을 설정한다.
2. 최초 접속자는 두 사람 이름을 설정한다.
3. 이후 접속자는 PIN을 입력해 장부에 들어간다.
4. 월별 대시보드 상단에서 전체 합계와 사람별 합계를 확인한다.
5. 요약 아래의 빠른 입력 폼에서 `쓴 사람`을 선택하고 `날짜`, `금액`, `용도`를 입력한다.
6. 모바일에서는 오른쪽 위 `상세내역` 버튼으로 내역 화면에 들어간다.
7. 상세 내역 화면에서 이번 달 내역을 확인하거나 다른 달을 선택해 살펴본다.

## 제품 범위

### 포함

- 단일 공동 장부
- 환경변수 기반 공유 PIN 접근 보호
- 월별 대시보드
- 사람별 지출 합계
- 지출 추가와 삭제
- 용도 자동 추천
- Docker 이미지와 Proxmox 배포용 실행 문서

### 제외

- 개인 계정/로그인
- 여러 가계부/초대 기능
- 지출 수정 기능
- 정산 계산
- 예산/카테고리/리포트
- 카드, 은행, OCR 연동

## 기술 설계

권장 구현은 Next.js 단일 앱과 PostgreSQL이다. 화면, API, 서버 DB 접근을 한 프로젝트에서 관리해 MVP 복잡도를 낮춘다. `DESIGN.md`의 Starbucks-inspired 스타일은 따뜻한 크림 배경, 진한 그린 포인트, 둥근 입력 컨트롤, 절제된 카드 그림자를 Paybook에 맞게 차용한다.

### 데이터 모델

- `household`: 단일 장부 설정. `person_a_name`, `person_b_name`, `created_at`
- `expense`: 지출. `id`, `spender`, `spent_on`, `amount`, `purpose`, `created_at`

`spender`는 설정된 두 이름 중 하나만 허용하며 UI에서는 직접 타이핑하지 않고 선택한다. `amount`는 양의 정수 원화 금액으로 저장한다. `purpose`는 자유 텍스트이며 최근/빈도 기반 추천에 사용한다. PIN은 DB에 저장하지 않고 서버 환경변수 `PAYBOOK_PIN`과 비교한다.

### 화면

- 초기 설정 화면: 두 사람 이름 입력
- PIN 잠금 화면: PIN 입력, 실패 메시지
- 데스크톱 월별 대시보드: 월 이동, 전체 합계, 사람별 합계, 사람 선택형 빠른 입력 폼, 최근/자주 쓴 용도 5개 추천, 상세 내역
- 모바일 기본 화면: 전체/사람별 합계와 지출 입력 폼을 우선 노출하고, 오른쪽 위 `상세내역` 버튼을 제공한다.
- 모바일 상세 내역 화면: 월 선택 컨트롤, 월 합계, 날짜순 내역, 삭제 액션을 제공한다.

## 데이터 흐름과 API

- 설정 여부 확인: 앱 진입 시 장부 설정 존재 여부를 확인한다.
- PIN 검증: 입력 PIN을 서버 환경변수 `PAYBOOK_PIN`과 비교하고 세션 쿠키를 발급한다.
- 월별 조회: 선택 월의 합계와 상세 내역을 함께 반환한다.
- 지출 추가: 입력 검증 후 DB에 저장하고 월별 데이터를 갱신한다.
- 지출 삭제: 해당 지출을 삭제하고 월별 데이터를 갱신한다.
- 용도 추천: 기존 `expense.purpose` 값에서 최근/빈도 기반 후보 5개를 반환한다.

## 환경과 보안

- 필수 환경변수는 `DATABASE_URL`, `PAYBOOK_PIN`, `PAYBOOK_SESSION_SECRET`이다.
- `DATABASE_URL`은 `postgresql://user:password@host:5432/database` 형식으로 PostgreSQL 서버 접속 정보를 담는다.
- `PAYBOOK_PIN`은 컨테이너 실행 시 주입하며 DB나 이미지에 저장하지 않는다.
- `PAYBOOK_SESSION_SECRET`은 PIN 검증 후 발급하는 세션 쿠키 서명에 사용한다.
- 개인용 배포라도 HTTPS 환경을 전제로 한다.
- URL이 외부에 노출될 수 있으므로 PIN 화면 전에는 어떤 지출 데이터도 반환하지 않는다.

## Docker와 Proxmox 배포

- 애플리케이션은 Dockerfile로 빌드 가능한 단일 이미지로 제공한다.
- Proxmox에서는 Docker가 설치된 VM 또는 LXC 안에서 컨테이너를 실행한다.
- PostgreSQL은 외부 서버를 사용하며, 앱 컨테이너에는 DB를 포함하지 않는다.
- 배포 예시는 `docker run` 또는 `docker compose` 기준으로 문서화한다.
- 컨테이너 포트는 기본 `3000`으로 열고, Proxmox 앞단의 리버스 프록시에서 HTTPS를 종료한다.

## 구현 순서

1. Next.js + TypeScript 앱 스캐폴딩
2. PostgreSQL 접근 계층과 환경 변수 예시 추가
3. Dockerfile과 컨테이너 실행 문서 추가
4. 이름 설정/PIN 세션 흐름 구현
5. 월별 대시보드와 빠른 입력 UI 구현
6. 지출 추가, 삭제, 월별 합계 API 구현
7. 용도 추천 구현
8. 빌드, 컨테이너 빌드, 핵심 흐름 검증

## 검증 계획

- 두 사람 이름 설정 후 PIN으로 진입할 수 있는지 확인한다.
- PIN 없이 API가 지출 데이터를 반환하지 않는지 확인한다.
- 잘못된 `PAYBOOK_PIN` 입력이 거부되는지 확인한다.
- 지출 추가 시 선택한 사람이 저장되고 전체 합계와 사람별 합계가 즉시 맞는지 확인한다.
- 용도 추천 5개 중 하나를 선택하거나 새 텍스트를 입력해 저장할 수 있는지 확인한다.
- 삭제 후 합계와 상세 내역이 갱신되는지 확인한다.
- 이번 달 기본값과 다른 월 이동이 동작하는지 확인한다.
- 모바일 기본 화면에서 상세 내역이 접혀 있고, `상세내역` 버튼을 눌러 월별 내역 화면으로 이동할 수 있는지 확인한다.
- 모바일 상세 내역 화면에서 다른 달 선택 후 해당 월 내역과 합계가 표시되는지 확인한다.
- Docker 이미지가 빌드되고 `DATABASE_URL`, `PAYBOOK_PIN`, `PAYBOOK_SESSION_SECRET` 주입만으로 실행되는지 확인한다.

## 미결 위험

- 로그인 없는 PIN 방식은 가족용 최소 보호 수준이다. URL과 PIN이 함께 유출되면 장부가 노출될 수 있다.
- Proxmox 자체는 앱 보안 경계가 아니므로 HTTPS, 방화벽, 리버스 프록시 설정이 필요하다.
- 단일 장부만 지원하므로, 여러 가계부가 필요해지면 데이터 모델을 확장해야 한다.
