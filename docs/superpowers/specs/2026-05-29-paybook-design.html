Paybook 공동가계부 설계

Docker 이미지로 배포해 Proxmox에서 사용하는 부부용 공동가계부 MVP.

문제 정의

Paybook은 로그인 없이 환경변수로 설정한 공유 PIN으로 접근하고, 누가 언제 어떤 용도로 얼마를 썼는지 남기는 단순한 공동가계부다. 최종 배포 형태는 Docker 이미지이며 Proxmox에서 컨테이너로 실행한다. 정산, 카드 연동, 영수증 OCR, 예산 관리는 MVP 범위에서 제외한다.

성공 기준

  • 첫 사용 시 두 사람의 이름을 설정한다.
  • PIN 입력 후 이번 달 전체 합계와 사람별 합계를 본다.
  • 오늘 날짜 기본값으로 지출을 빠르게 추가한다.
  • 쓴 사람은 두 이름 중 하나를 버튼/콤보박스 형태로 선택한다.
  • 최근/자주 사용한 용도 5개를 추천하고, 직접 입력도 허용한다.
  • 모바일 기본 화면은 지출 입력을 중심에 두고, 오른쪽 위 상세내역 버튼으로 월별 내역을 본다.
  • 상세 내역 화면에서는 날짜순 내역을 확인하고, 다른 달도 선택해서 살펴본다.

사용자 흐름

  1. 배포자는 컨테이너 환경변수로 PostgreSQL 접속 정보와 공유 PIN을 설정한다.
  2. 최초 접속자는 두 사람 이름을 설정한다.
  3. 이후 접속자는 PIN을 입력해 장부에 들어간다.
  4. 월별 대시보드 상단에서 전체 합계와 사람별 합계를 확인한다.
  5. 요약 아래의 빠른 입력 폼에서 쓴 사람을 선택하고 날짜, 금액, 용도를 입력한다.
  6. 모바일에서는 오른쪽 위 상세내역 버튼으로 내역 화면에 들어간다.
  7. 상세 내역 화면에서 이번 달 내역을 확인하거나 다른 달을 선택해 살펴본다.

포함 범위

  • 단일 공동 장부
  • 환경변수 기반 공유 PIN 접근 보호
  • 월별 대시보드와 사람별 합계
  • 지출 추가, 삭제, 용도 추천
  • Docker 이미지와 Proxmox 배포용 실행 문서

제외 범위

  • 개인 계정/로그인
  • 여러 가계부/초대 기능
  • 수정, 정산, 예산, 카테고리
  • 카드, 은행, OCR 연동

기술 설계

Next.js 단일 앱과 PostgreSQL을 사용한다. 화면, API, 서버 DB 접근을 한 프로젝트에서 관리해 MVP 복잡도를 낮춘다. DESIGN.md의 따뜻한 크림 배경, 진한 그린 포인트, 둥근 입력 컨트롤, 절제된 카드 그림자를 Paybook에 맞게 차용한다.

데이터 모델

  • household: person_a_name, person_b_name, created_at
  • expense: id, spender, spent_on, amount, purpose, created_at

spender는 직접 타이핑하지 않고 설정된 두 이름 중 하나를 선택한다. purpose는 자유 텍스트이며 최근/빈도 기반 추천에 사용한다. PIN은 DB에 저장하지 않고 서버 환경변수 PAYBOOK_PIN과 비교한다.

API와 데이터 흐름

  • 설정 여부 확인 후 초기 설정 또는 PIN 화면으로 보낸다.
  • PIN은 서버 환경변수 PAYBOOK_PIN과 비교하고 세션 쿠키를 발급한다.
  • 월별 조회는 합계와 상세 내역을 함께 반환한다.
  • 지출 추가/삭제 후 월별 데이터를 갱신한다.
  • 용도 추천은 기존 지출의 최근/빈도 기반 후보 5개를 반환한다.

모바일 화면 정책

  • 기본 모바일 화면은 지출 입력을 주 용도로 하며 상세 리스트를 길게 노출하지 않는다.
  • 오른쪽 위 상세내역 버튼을 누르면 월별 내역 화면으로 이동한다.
  • 상세 내역 화면은 월 선택 컨트롤, 월 합계, 날짜순 내역, 삭제 액션을 제공한다.

환경과 보안

필수 환경변수는 DATABASE_URL, PAYBOOK_PIN, PAYBOOK_SESSION_SECRET이다. DATABASE_URL은 PostgreSQL 접속 문자열이며, PAYBOOK_PIN은 컨테이너 실행 시 주입한다. PIN 화면 전에는 어떤 지출 데이터도 반환하지 않는다.

Docker와 Proxmox 배포

  • 애플리케이션은 Dockerfile로 빌드 가능한 단일 이미지로 제공한다.
  • Proxmox에서는 Docker가 설치된 VM 또는 LXC 안에서 컨테이너를 실행한다.
  • PostgreSQL은 외부 서버를 사용하며, 앱 컨테이너에는 DB를 포함하지 않는다.
  • 컨테이너 포트는 기본 3000으로 열고, 리버스 프록시에서 HTTPS를 종료한다.

구현 순서

  1. Next.js + TypeScript 앱 스캐폴딩
  2. PostgreSQL 접근 계층과 환경 변수 예시 추가
  3. Dockerfile과 컨테이너 실행 문서 추가
  4. 이름 설정/PIN 세션 흐름 구현
  5. 월별 대시보드와 빠른 입력 UI 구현
  6. 지출 추가, 삭제, 월별 합계 API 구현
  7. 용도 추천 구현
  8. 빌드, 컨테이너 빌드, 핵심 흐름 검증

검증 계획

  • 두 사람 이름 설정 후 PIN으로 진입할 수 있는지 확인한다.
  • PIN 없이 API가 지출 데이터를 반환하지 않는지 확인한다.
  • 잘못된 PAYBOOK_PIN 입력이 거부되는지 확인한다.
  • 선택한 사람이 저장되고 지출 추가/삭제 후 합계와 상세 내역이 갱신되는지 확인한다.
  • 용도 추천 5개 중 하나를 선택하거나 새 텍스트를 입력해 저장할 수 있는지 확인한다.
  • 이번 달 기본값과 다른 월 이동이 동작하는지 확인한다.
  • 모바일 기본 화면에서 상세 내역 버튼으로 월별 내역 화면에 들어갈 수 있는지 확인한다.
  • 모바일 상세 내역 화면에서 다른 달 선택 후 해당 월 내역과 합계가 표시되는지 확인한다.
  • Docker 이미지가 빌드되고 환경변수 주입만으로 실행되는지 확인한다.

미결 위험

PIN 방식은 가족용 최소 보호 수준이다. URL과 PIN이 함께 유출되면 장부가 노출될 수 있다. Proxmox 자체는 앱 보안 경계가 아니므로 HTTPS, 방화벽, 리버스 프록시 설정이 필요하다. 단일 장부만 지원하므로 여러 가계부가 필요하면 데이터 모델 확장이 필요하다.
